# DBil v0.1 — Plan 5 (Frontend skeleton)

## Context

Plans 1–4 made every backend capability addressable through curl. Plan 5
embeds the first browser UI on top of that API so DBil becomes something a
user can actually *use*. After Plan 5 a person can open `localhost:4242`,
log in, register a Postgres connection, run a query, and see the result —
no curl required.

Out of scope for Plan 5: ER diagram (Plan 7), schema browser (Plan 7),
observability dashboards (Plan 6), query history, saved queries,
multi-tab editor. They each deserve their own plan; cramming them here
would dilute focus on the foundation everyone else builds on.

## Spec alignment

Implements spec section 4 (frontend stack), section 5 (browser → HTTP
server flow), and the strict CSP / same-origin parts of section 6.6.
React + Vite are the locked choices.

## Tasks

### Phase A — Frontend scaffold

**Task 1** — Initialize `web/`:
- `cd web && npm init -y && npm i react react-dom react-router-dom @tanstack/react-query @tanstack/react-table @uiw/react-codemirror @codemirror/lang-sql @codemirror/state @codemirror/view`
- Dev deps: `vite @vitejs/plugin-react typescript @types/react @types/react-dom @types/node tailwindcss postcss autoprefixer`
- `vite.config.ts` with `base: "/"`, dev proxy `"/api"` → `http://localhost:4242`, build output `dist/`.
- `tsconfig.json` strict, `target: ES2022`, `jsx: react-jsx`.
- `tailwind.config.js` content globs over `src/**/*.{ts,tsx}`.
- `index.html` with `<div id="root">`.
- `src/main.tsx` mounts `<App />` inside `QueryClientProvider` and `BrowserRouter`.
- `src/styles.css` only contains `@tailwind base;@tailwind components;@tailwind utilities;` + a `:root` dark-friendly palette.
- `src/App.tsx` with routes: `/` (Editor or redirect to login), `/login`, `/connections`.

**Task 2** — `.gitignore`:
- `web/node_modules/`
- `web/dist/*` with `!web/dist/.gitkeep`
- Commit `web/dist/.gitkeep` so `go:embed` always finds at least one file.

### Phase B — Go embed + SPA serving

**Task 3** — `web/embed.go` (in `package web`):
```go
package web

import "embed"

//go:embed all:dist
var DistFS embed.FS
```

**Task 4** — `internal/server/handlers/spa.go`:
- `func SPAHandler() http.Handler` reads from `web.DistFS`, strips the
  `dist/` prefix via `fs.Sub`, falls back to `index.html` for unmatched
  paths (so client-side routes like `/connections` reload-safe).
- Strict response headers: `Content-Security-Policy: default-src 'self';
  script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self'
  data:; connect-src 'self'; frame-ancestors 'none'`, `X-Content-Type-Options:
  nosniff`, `Referrer-Policy: no-referrer`.
- Skips `/api/*` and `/healthz` (those routes are already mounted higher
  in the chi tree; chi tries the static handler last via `r.NotFound`).

**Task 5** — `handlers.Mount`:
- Register the SPA handler via `r.NotFound(SPAHandler().ServeHTTP)` so it
  catches everything chi's routes don't match.
- `lint-auth` is unaffected — the SPA is not an API route and lives outside
  the RequireAuth tree by design (the React side calls `/api/auth/login`
  unauthed, then proves auth on every API call via `Authorization: Bearer`).

**Task 6** — `spa_test.go`:
- Build the handler with the embedded FS.
- Assert `GET /` returns `200` with a body containing the dist `index.html`
  string. (Plan-5 test uses a placeholder index.html committed in
  `web/dist/index.html`; a real built bundle replaces it at deploy time.)
- Assert `GET /not-a-real-route` falls back to `index.html` (200, not 404)
  so client-side routes reload-safe.
- Assert security headers are present.

### Phase C — Auth (login + token store)

**Task 7** — `web/src/api/client.ts`:
- `apiFetch(path, options)` wraps `fetch` with base URL `""`, attaches
  `Authorization: Bearer ${token}` from localStorage when present,
  throws on non-2xx with a typed `ApiError`.

**Task 8** — `web/src/auth/AuthContext.tsx`:
- `useAuth()` returns `{token, user, login(email, pw), logout()}`. Token
  persisted in `localStorage["dbil_token"]`. On mount, fetches `/api/me`
  to validate; clears on 401.

**Task 9** — `web/src/pages/LoginPage.tsx`:
- Centered form, email + password fields, submit POSTs to `/api/auth/login`.
- On success calls `useAuth().login(...)`, navigates to `/connections`.
- On 401 shows "Invalid credentials"; on 5xx shows "Server error".

**Task 10** — `web/src/components/ProtectedRoute.tsx`:
- Reads auth context; if no token, redirects to `/login`.

### Phase D — Connections UI

**Task 11** — `ConnectionsPage.tsx`:
- React Query `useQuery(["connections"])` → `GET /api/connections`.
- Renders a table: alias, host:port, tag (with `<TagBadge>`), test button,
  delete button, "open in editor" link.
- "Add connection" opens `<ConnectionFormDialog>`.

**Task 12** — `ConnectionFormDialog.tsx`:
- Modal with all fields (alias, host, port, username, password, database,
  tag select, tls_mode select, optional passphrase).
- On submit POST `/api/connections`; on 409 shows duplicate-alias error.

**Task 13** — `TagBadge.tsx`:
- Renders a small coloured pill; production = red, staging = amber,
  dev = green, local = gray. The contrast keeps the production warning
  visible without screaming.

### Phase E — Editor UI

**Task 14** — `EditorPage.tsx`:
- Sidebar with connection list, current selection highlighted.
- Top bar: tag badge + "Run" button (Cmd/Ctrl+Enter).
- Middle: CodeMirror with `@codemirror/lang-sql`.
- Bottom: results grid (TanStack Table). Columns derived from response
  `columns`; rows from `rows`. Sticky header, virtualized rows.
- Production-tagged sessions get a confirm-checkbox above Run that, when
  ticked, sends `X-Confirm: yes`.
- Errors (4xx / 5xx) render below the editor as a red banner with the
  server-reported reason.

### Phase F — Build integration

**Task 15** — Makefile additions:
- `web-deps:` runs `cd web && npm ci` (or `npm install` if no lockfile).
- `web-build:` runs `cd web && npm run build`.
- `build:` now depends on `web-build` so a fresh checkout produces a
  binary with a real bundle.

**Task 16** — `Dockerfile`:
- Add a Node build stage `--platform=$BUILDPLATFORM node:22-alpine AS web`
  that copies `web/` and runs `npm ci && npm run build`.
- Go build stage `COPY --from=web /src/web/dist ./web/dist` before the
  `go build` line so the embed directive finds real assets.

**Task 17** — README update:
- Mention `make web-build` for contributors hacking on the UI.
- Snippet of the docker-compose drop-in that brings up Postgres + DBil
  and opens the UI on `:4242`.

### Phase G — Smoke + tag

**Task 18** — Local smoke:
- `make web-build && make build && ./bin/dbil version`.
- `DBIL_DATA_DIR=$(mktemp -d) ./bin/dbil init` + serve, open `localhost:4242`,
  log in with the printed credentials, add a connection, run `SELECT 1`.
- Confirmation: results render in the grid.

**Task 19** — Tag `v0.5.0-frontend`.

## Risks acknowledged

- **CSP `style-src 'self' 'unsafe-inline'`:** Tailwind's runtime classes
  don't need inline styles, but CodeMirror does (it injects styles
  inline). The spec said no `unsafe-inline`; this plan documents the
  exception. A future pass can move CodeMirror styles to a stylesheet
  + nonces.
- **First-load JS budget:** the spec set a frontend goal of ≤200KB
  gzipped. CodeMirror + TanStack alone push that; Plan 5 lands ~280KB.
  The optimisation (lazy-loading the editor, splitting routes) goes into
  a v0.5.1 pass once the rest is in.
- **No tests on the React side yet.** Plan 5 keeps testing pragmatic:
  Go-side tests cover the SPA serving + headers, and the e2e plan from
  Plan 4 still proves the API contract. React component tests arrive
  with Plan 6 when the observability UI introduces more logic that
  warrants them.
- **No HTTPS termination in the binary.** README continues to recommend
  Caddy / Traefik in front. Plan 5 does not change that.
