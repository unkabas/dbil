Прямо в твой docker-compose файл.

Один контейнер, React-интерфейс уже внутри, ~20 МБ на диске. Граф схем, редактор запросов, живые метрики, цепочка аудита с защитой от подделки. По умолчанию отказывает в DDL на соединениях с тегом production.

Быстрый старт На GitHub

ghcr.io/unkabas/dbil:latest · multi-arch · подпись cosign

Рабочее место, а не админка.

Подключаешь dbil к Postgres и получаешь граф схем, собранный прямо из pg_catalog, редактор SQL с автодополнением, пейджер строк, который реально удобен, и вкладку с живыми метриками. Соединения с тегом production отказывают в DDL; DML требует заголовка X-Confirm. Всё, ради чего ты обычно пишешь обёртку-скрипт, dbil уже делает.

  • образ ~20 МБ · подпись cosign · multi-arch
  • арх linux + darwin · amd64 + arm64
  • состояние SQLite в /data/dbil.db, шифрование AES-256-GCM
  • ui React, встроен, без CDN в рантайме
  • сеть обнаружение через docker-лейблы, без открытых портов
  • лицензия Apache 2.0 · без телеметрии · без трекинга

Слушай, как дышит база.

TPS, cache-hit, лаг репликации, медленные запросы, цепочки блокировок, неиспользуемые индексы. Если запрос блокирует пять других сессий, жми Kill, и dbil отправит pg_terminate_backend. Советчик читает pg_stat_user_indexes и показывает, что можно убрать.

Чего в dbil нет.

Каждая отсутствующая фича — осознанный отказ. Контейнер остаётся маленьким, интерфейс — читаемым, установка — одним сервисом в твоём compose-файле именно благодаря этим отказам, а не вопреки им.

  • × нет телеметрии образ никогда не звонит наружу.
  • × нет облачной синхронизации состояние живёт на твоём диске, зашифрованное.
  • × нет AI-моста для чата редактор не пишет SQL за тебя. Пиши запрос сам.
  • × нет UI прав для команд роли живут в Postgres-соединении, а не в dbil.

Отказывает в том, чего ты не имел в виду.

Тег соединения — это контракт. Три отказа держат вес; остальное — в документации.

01

Production отказывает на сетевом уровне.

DDL блокируется. DML требует X-Confirm: yes. DELETE/UPDATE без WHERE отказаны выше local.

HTTP 412
02

Учётки в plaintext: отказ.

Свой DEK на каждое соединение, AES-256-GCM. Шесть лоадеров мастер-ключа; env-var печатает предупреждение.

env var ⚠
03

Подделка видна на первой же проверке.

SHA-256 хеш цепочки на каждой строке аудита. VerifyChain ловит мутацию мгновенно.

CI-gated

Прямо в твой compose.

Два лейбла на сервис Postgres, общая сеть, dbil с примонтированным Docker-сокетом. На первом запуске пароль администратора уходит в логи и в файл /data/initial-credentials.txt. 

services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: appdb
    labels:
      dbil.enable: "true"
      dbil.alias:  "App DB"
      dbil.tag:    "dev"
      dbil.creds.username_env: "POSTGRES_USER"
      dbil.creds.password_env: "POSTGRES_PASSWORD"
      dbil.creds.database_env: "POSTGRES_DB"
    networks: [appnet]

  dbil:
    image: ghcr.io/unkabas/dbil:latest
    command: ["serve"]
    ports: ["4242:4242"]
    volumes:
      - dbil_data:/data
      - /var/run/docker.sock:/var/run/docker.sock:ro
    environment:
      DBIL_DISCOVER: "docker"
      DBIL_NETWORK:  "appnet"
    networks: [appnet]

volumes: { dbil_data: }
networks: { appnet: { name: appnet } }
# задай пароль для postgres
echo "POSTGRES_PASSWORD=$(openssl rand -base64 24)" > .env

# старт; пароль администратора уходит в логи dbil
docker compose up -d
# готовые бинарники для linux/{amd64,arm64} и darwin/{amd64,arm64}
DBIL_DATA_DIR=./dbil-data \
  ./dbil serve

Проверь перед запуском.

Каждый релиз-тег публикует multi-arch образ, подписанный cosign keyless OIDC. SPDX SBOM прикрепляется к каждому релизу на GitHub. Файл состояния (/data/dbil.db) зашифрован на уровне приложения: утёкший .db остаётся шифротекстом без мастер-ключа.
