Quick start.

dbil is one ~20 MB container with the React UI baked in. There are two install paths: drop it into an existing docker-compose.yml for auto-discovery, or run it standalone and add connections by hand. Pick the one that fits where your Postgres already lives.

Quick start

The two paths are not better-or-worse; they fit different setups. Option A is for projects that already use compose and want dbil to read database credentials from container labels. Option B is for everyone else, including managed-cloud Postgres, a database on your laptop, or a remote dev VM.

Option A: auto-discovery via compose labels

Add two things on your Postgres service: a set of dbil.* labels, and a shared network. Then add dbil to that network with the Docker socket mounted read-only. dbil scans for containers carrying its labels and opens the connections for you. No manual credential entry.

Then bring it up. The first run writes the admin password to /data/initial-credentials.txt inside the dbil container and also prints it to the logs.

Open http://localhost:4242, sign in with the admin password, and you should see the App DB connection already listed in the sidebar with the dev tag.

why ro The Docker socket is mounted :ro because dbil only reads container labels and resolves env-var values. It never starts, stops, or modifies containers.

Option B: run standalone, connect in the UI

Best when your Postgres lives somewhere dbil can’t see labels on: a managed cloud, your laptop, a remote dev VM, or a separate compose project. Run dbil by itself, then add connections through the workspace UI.

Open http://localhost:4242, read the admin password from the logs (docker logs dbil), then jump to Adding a connection below.

Verify the image

Every release tag publishes a multi-arch image signed with cosign keyless OIDC. SPDX SBOMs are attached to each GitHub release.

First-run admin password

On the very first start, dbil generates an admin password and writes it to two places:

  • the dbil logs (one line, prefixed admin password:)
  • /data/initial-credentials.txt inside the container (so it survives a log rotation)

The file is deleted automatically the first time you successfully sign in. To rotate, sign out and run docker exec dbil dbil reset-admin.

Adding a connection

If you took Option A, connections already appear in the sidebar. The flow below is for Option B, or for any second database you add later regardless of install path.

  1. Click Connections in the left sidebar.
  2. Click Add connection in the panel header.
  3. Fill in host, port, database, user, password.
  4. Pick a tag: local, dev, staging, or production. The tag is the contract that decides what dbil refuses; see Tags & production safety.
  5. Click Test. dbil opens the connection, runs SELECT 1, and reads pg_catalog. Green means ready.
  6. Save. The password is immediately encrypted with a per-connection DEK derived from the master key and written to /data/dbil.db as ciphertext. It never sits in plaintext on disk.

On production-tagged connections, dbil also asks for a separate passphrase that is not derived from the master key. That passphrase is required every time you open the connection in a new session.

passphrase Lose the passphrase, lose access. There is no recovery flow because there is no path that would not also be an attack path. Treat production passphrases the way you treat the database backup encryption key.

Tags & production safety

Every connection carries a tag. The tag decides which statements dbil refuses before they leave the browser, what the workspace prompts for, and what gets logged.

tag DDL DML DELETE / UPDATE without WHERE
local allowed allowed allowed
dev allowed X-Confirm refused
staging refused X-Confirm refused
production refused X-Confirm + passphrase refused

Refused means the request returns HTTP/1.1 412 Precondition Required from the dbil server before it ever reaches Postgres. The browser shows the reason inline; the rejected statement is logged to the audit chain with the tag attached.

X-Confirm means the DML statement is held until the user explicitly confirms it through the editor (which sets an X-Confirm: yes header on the request). The confirmation does not persist between statements.

Statement timeouts are tag-driven: five minutes on local, thirty seconds on dev, ten seconds above. To change them, edit /data/dbil.db via dbil config edit.

Troubleshooting

The dbil container starts but no connections appear (Option A)

dbil only discovers containers on the network named in DBIL_NETWORK. If your Postgres service is on a different network, either add it to appnet or set DBIL_NETWORK to the matching one. Confirm with docker network inspect appnet.

I lost the admin password before I signed in

The password is at /data/initial-credentials.txt until you sign in once. After that: docker exec dbil dbil reset-admin.

Connection test returns “permission denied for table pg_class”

The Postgres user needs read on pg_catalog. Grant pg_read_all_data or use a role with explicit SELECT on information_schema.

Schema graph is empty after connecting

dbil scans only schemas listed in the connection’s visible schemas field (default public). Edit the connection to add more, then click Refresh schema.
