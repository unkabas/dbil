ghcr.io/unkabas/dbil:latest · multi-arch · cosign-signed

A workspace, not an admin tool.

Point dbil at a Postgres instance and you get a schema graph built straight from pg_catalog, a SQL editor with autocompletion, the row pager you actually want, and a live observability tab. Production‑tagged connections refuse DDL outright; DML wants an X-Confirm header. Everything you’d write a wrapper script for, dbil already does.

  • image ~20 MB · cosign-signed · multi-arch
  • arch linux + darwin · amd64 + arm64
  • state SQLite at /data/dbil.db, AES-256-GCM envelope
  • ui React, baked in, no CDN at runtime
  • network discovers via docker labels, no exposed ports
  • license Apache 2.0 · no telemetry · no tracking

Watch the database breathe.

TPS, cache‑hit ratio, replication lag, slow queries, lock chains, unused indexes. If something is blocking five other sessions, hit Kill and dbil sends pg_terminate_backend. The advisor reads pg_stat_user_indexes and tells you what to drop.

What dbil doesn’t ship.

Every absent feature is a deliberate cut. The container stays small, the surface stays legible, and the install stays one service in your compose file because of these refusals, not in spite of them.

  • × no telemetry the image phones home to nothing, ever.
  • × no cloud sync state lives on your disk, encrypted at rest.
  • × no AI chat bridge the editor doesn’t write SQL for you. Bring your own query.
  • × no team-RBAC UI roles live in the Postgres connection, not in dbil.

Refuses what you didn’t mean.

A connection’s tag is a contract. Three refusals carry the weight; the rest is in the docs.

01

Production refused at the wire.

DDL blocked. DML needs X-Confirm: yes. DELETE/UPDATE without WHERE refused above local.

HTTP 412
02

Plaintext credentials, refused.

Per-connection DEK, AES-256-GCM. Six master-key loaders; env-var prints a warning.

env var ⚠
03

Tamper shows up the first scan.

SHA-256 chain hash on every audit row. VerifyChain flags mutation immediately.

Drop it into your compose.

Two labels on your Postgres service, a shared network, dbil with the Docker socket. First run prints the admin password to the logs and to /data/initial-credentials.txt. 

services:
              postgres:
                image: postgres:16
                environment:
                  POSTGRES_USER: app
                  POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
                  POSTGRES_DB: appdb
                labels:
                  dbil.enable: "true"
                  dbil.alias:  "App DB"
                  dbil.tag:    "dev"
                  dbil.creds.username_env: "POSTGRES_USER"
                  dbil.creds.password_env: "POSTGRES_PASSWORD"
                  dbil.creds.database_env: "POSTGRES_DB"
                networks: [appnet]

              dbil:
                image: ghcr.io/unkabas/dbil:latest
                command: ["serve"]
                ports: ["4242:4242"]
                volumes:
                  - dbil_data:/data
                  - /var/run/docker.sock:/var/run/docker.sock:ro
                environment:
                  DBIL_DISCOVER: "docker"
                  DBIL_NETWORK:  "appnet"
                networks: [appnet]

            volumes: { dbil_data: }
            networks: { appnet: { name: appnet } }
# set a postgres password
            echo "POSTGRES_PASSWORD=$(openssl rand -base64 24)" > .env

            # start it; admin password prints to the dbil logs
            docker compose up -d
# prebuilt for linux/{amd64,arm64} and darwin/{amd64,arm64}
            DBIL_DATA_DIR=./dbil-data \
              ./dbil serve

Verify before you run.

Every release tag publishes a multi‑arch image signed with cosign keyless OIDC. SPDX SBOMs are attached to each GitHub release. The state file (/data/dbil.db) is application‑encrypted: a leaked .db file is still ciphertext without the master key.
